  • AI as a system component, not a feature

    Getting data into our enterprise metrics platform required manual SQL, manual YAML, and careful human precision at every step. When upstream metric logic changed, teams redid quarters of work — one team lost an entire quarter to exactly that. Strong data engineers were functioning as careful SQL and YAML editors. That's not a staffing problem; it's a systemic design failure. Those are different mandates.

    I mandated an engineering solution with explicit constraints: humans stay accountable for correctness and meaning, automation reduces work without obscuring logic, and the system is held to production standards — not built as a demo. My team delivered a RAG-grounded pipeline: a natural language question in, hybrid keyword + vector retrieval over an Elasticsearch index of approved schemas, LLM-generated SQL with automated validation — schema correctness, join validity, NULL handling, a confidence score — then extraction of governed measures and dimensions, standards-compliant YAML, and an automated pull request into the governed metrics repository, all surfaced through a Backstage developer portal plugin.

  • It wasn't a fraud system. It was an adversarial identity graph.

    I was hired to replace two rigid, monolithic fraud web services sitting in the live booking path — the only system standing between Expedia and many millions of dollars in fraud losses. Once fraudsters detect an outage, losses escalate exponentially within hours.

    What we actually built was a real-time identity resolution platform operating under adversarial conditions, where signals actively try to evade correlation. We decomposed the monoliths into a modular platform: signal collection from more than a dozen parallel sources, a stateful behavioral graph in Elasticsearch aggregating velocity over time windows, deterministic known-entity lists, ML confidence scoring, an explicit rules engine gating merge decisions, and full decision-context persistence so every outcome could be audited and explained.     Fraud losses fell 10% year over year, and the platform absorbed supplier fraud and account takeover workloads without relaxing a single SLA. Errors in either direction had real cost — false merges let fraud through, false splits broke legitimate customers — which is what forced the explicit confidence thresholds and conservative merge logic that made the system trustworthy.

  • Petabyte scale with no safety net

    At SDL I owned the complete product end-to-end in a private data center: cluster architecture, storage, capacity planning, ingestion, enrichment, indexing, query serving — and the application layer on top, down to a pixel-perfect UI. No managed services. No elastic scaling. Every capacity decision and failure mode was ours.

    The architecture was what would later be named a Lambda Architecture, built before the pattern had a name: Kafka as the ingestion bus absorbing viral social-media traffic spikes, HBase as the durable write store and MapReduce input surface, Oozie- orchestrated batch jobs running ML enrichment — sentiment analysis, entity extraction, language detection at batch scale — and date-partitioned Elasticsearch as the query surface for client reporting.

    The modern equivalents are managed: Kinesis, S3, Airflow, OpenSearch, feature stores. The engineering discipline is identical — we just paid for mistakes in recovery time instead of cloud spend.

  • Migrating a petabyte, live, exactly once

    Our HBase row keys were timestamp-prefixed, which meant every incoming write concentrated on the single region server that owned the current time range while the rest of the cluster sat idle. Under production load, that's a ceiling you hit hard.

    The fix was well understood — salt the keys with a hash prefix to spread writes evenly across regions. The problem was that HBase row keys are immutable: fixing the design meant rewriting the entire corpus. At petabyte scale. On live infrastructure. In a private data center with no elastic compute to lean on.

    We executed it as chunked Oozie workflows with state-aware restart logic and dual-table consistency validation before cutover. The tradeoff — time-range scans became parallel scans across salted partitions — was priced in and worth it. Planned carefully, executed correctly, never repeated.
